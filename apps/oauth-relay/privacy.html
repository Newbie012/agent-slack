Privacy

agent-slack is a local command-line tool with no backend. It does not receive or store your Slack data. Slack content is read directly by the operator's machine and is not persisted by the project.

Data flow

  • Slack tokens are exchanged directly between the operator's machine and Slack, and stored only on that machine (macOS Keychain, or a 0600 file on other platforms). They are never sent to this relay or any project server.
  • Slack content (messages, users, files) is read directly from Slack by the operator's machine and returned to their terminal or agent. The project stores none of it.
  • During login, this relay receives Slack's OAuth code and state and immediately forwards them to the operator's localhost callback. It exists only because Slack requires an HTTPS redirect.

What this relay stores

Nothing. The relay is stateless, holds no secrets, never receives tokens, and sends cache-control: no-store. Login can also bypass the relay entirely using a localhost redirect.

Slack access

The CLI requests read-only scopes and can access only the Slack data allowed by the approved scopes, the operator's token type, channel membership, and workspace policy. It requests no write, admin, or destructive scopes.

Third parties and retention

No Slack data is stored on any server, sold, or shared with third parties. If the operator connects agent-slack to an AI agent, the content they read is passed to that agent's model provider, which is the operator's choice.

Revoking access

Workspace admins can revoke the app from Manage Apps at any time. Operators can run agent-slack auth logout, which revokes the token on Slack.

More detail and contact

Full security details and the threat model are in the project documentation. For questions or responsible disclosure, use GitHub Issues.
