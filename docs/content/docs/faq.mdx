## Do I need to create a Slack app?

No. `agent-slack auth login` uses a bundled public Slack app and PKCE, so the
default flow works with nothing but a browser. You only need your own app
credentials for [self-hosted OAuth](/docs/authentication#self-hosted-oauth) or
development against a specific app.

## Can I use it across multiple workspaces?

Yes. Each login becomes a named profile. Create one per workspace and select it
with `--profile`:

```bash
agent-slack auth login --profile acme
agent-slack conversation history C123 --profile acme --json
```

## Why a CLI instead of an MCP server or a browser tool?

A CLI is the smallest stable surface an agent can call: one binary, deterministic
JSON in and out, real exit codes, and no session state to manage. It runs the
same from a shell, a script, a CI job, or inside an agent's tool loop, and
you can pipe it, log it, and diff it like any other command.

It is also cheaper per task. An MCP server keeps its tool definitions resident in
the model's context, spending tokens before any work happens. The CLI is invoked
like any command, discovered on demand with `describe --json`, and returns slim,
compact output (see [token-efficient output](/docs/reference/output-contract)),
so an agent spends far fewer tokens to get the same Slack context.

## Does it work in CI or on a headless server?

Yes. Non-TTY stdout defaults to JSON, and you can authenticate without a browser
using an existing token or the printed OAuth URL. See
[Headless Auth](/docs/recipes/headless-auth).

## A command returns nothing. Is it broken?

Probably not. It's likely the [Slack boundary](/docs#the-slack-boundary).
Check the active profile and its scopes with `agent-slack auth status --json`,
confirm the token type is right for the data (`--token user` vs `--token bot`),
and make sure the user or bot can actually see that conversation. Real failures
come back as [typed errors](/docs/reference/troubleshooting), not empty output.

## Can it post or delete messages?

No. The scopes it requests are read-only, so the token itself cannot post, edit,
or delete. Slack rejects any write for lack of scope, regardless of the command.
As a second safeguard, the generic `api call` command also blocks known write,
destructive, and admin methods unless an operator explicitly passes
`--allow-write --yes`. (If an operator supplies their own token that has write
scopes via `--token`, that token's scopes govern.)

## Can it read channels, DMs, or files I don't have access to?

No. The token acts as the authorizing user (or bot), so the CLI can only retrieve
what that identity can already see in Slack. Slack enforces channel membership,
granted scopes, and workspace and admin policy on its side, so private channels,
group DMs, or direct messages you are not part of stay inaccessible. There is no
token that means "read all of Slack."

## Where are my tokens stored?

Locally, never on any server. On macOS the token is stored in the **Keychain**
(encrypted at rest); plaintext file storage is not allowed there. On other
platforms it lives in a `0600`-permission file at
`~/.config/agent-slack/profiles.json`. See [Security](/docs/security).

## Does my Slack data go to your servers?

No. agent-slack is a local CLI with no backend. It reads Slack directly from your
machine and stores none of your data. The hosted login relay only forwards the
OAuth `code` during sign-in and never sees your token. See [Privacy](/docs/privacy).

## How do I revoke access?

Run `agent-slack auth logout`, which revokes the token on Slack and removes the
local profile. Workspace admins can also revoke the app from Manage Apps at any
time.

## Is it safe to connect this to an AI agent?

Yes, with the usual care for any credential. The CLI is read-only and never
prints the token, and it treats Slack content as data, not instructions. Since
any agent you authorize can use the token, don't let the agent act on
instructions found in Slack content, and limit its shell and network access. Even
if a token did leak, it is bounded: read-only, expiring under token rotation, and
revocable. See [Security](/docs/security#credential-handling-and-prompt-injection).

## I'm a workspace admin. What should I know before approving?

It requests read-only scopes (no write, admin, or destructive access), runs
locally with no backend, stores nothing on a server, and is revocable at any
time. See [Security](/docs/security#at-a-glance).
