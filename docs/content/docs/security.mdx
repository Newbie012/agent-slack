agent-slack follows a zero-trust policy for its login infrastructure. The only
trusted parties are your local machine and Slack. The hosted OAuth relay is
treated as untrusted.

## For workspace admins

Reviewing agent-slack for your workspace? The short version:

- **Read-only.** Every requested scope is a read scope. The app cannot post,
  edit, delete, or take admin actions. The generic `api call` escape hatch blocks
  known write and admin methods unless an operator explicitly passes
  `--allow-write --yes`.
- **No backend, no data retention.** agent-slack is a local CLI. There is no
  server that receives or stores your Slack data. Slack content is fetched
  directly by the operator's machine, used in the moment, and not persisted by
  the project.
- **Tokens stay local.** OAuth tokens are stored on the operator's machine (macOS
  Keychain, or a `0600` file on other platforms), never on project
  infrastructure. The hosted OAuth relay never receives tokens.
- **Least privilege and revocable.** Access is bounded by the scopes you approve,
  the operator's token type, channel membership, and workspace policy. You can
  revoke it any time from Manage Apps, and `agent-slack auth logout` revokes the
  token on Slack.

### Scopes requested, and why

All are read scopes:

| Scope | Purpose |
| --- | --- |
| `channels:read`, `groups:read`, `im:read`, `mpim:read` | List public channels, private channels, DMs, and group DMs and their metadata so an operator can choose what to read. |
| `channels:history`, `groups:history`, `im:history`, `mpim:history` | Read message history in conversations the operator can access. |
| `users:read` | Resolve user IDs to display names so output is human-readable. |
| `files:read` | List and read metadata for files shared in accessible conversations, and download a file the operator explicitly requests. |
| `reactions:read`, `pins:read`, `bookmarks:read` | Read reactions, pinned items, and bookmarks as channel context. |
| `team:read` | Read basic workspace metadata (name, domain, enterprise id). |

The app requests no write, admin, or destructive scopes.

### What leaves the workspace

The project sends nothing to third parties. Slack API calls go directly from the
operator's machine to Slack. If the operator connects agent-slack to an AI agent,
the Slack content they read is passed to that agent's model provider, which is
the operator's own choice and outside this project. See
[Credential handling and prompt injection](#credential-handling-and-prompt-injection).

## Trust boundaries

Browser login uses PKCE. Your machine generates a `code_verifier`, sends it only
to Slack during the token exchange, and never sends it to the relay. The token
exchange and every rotation refresh happen directly between the CLI and Slack.

```
Slack  --code+state-->  relay (aslk.vercel.app)  --code+state-->  localhost CLI
                         (no verifier, no token,                       |
                          stateless, no secret)                        v
                                                       CLI  <--token-->  Slack
                                                       (code_verifier + client_id;
                                                        relay never involved)
```

What this means:

- **The relay never sees your tokens.** Access tokens and refresh tokens are
  exchanged and refreshed directly with Slack.
- **The relay cannot exchange an honest login's code.** Without the local
  `code_verifier`, a captured code is useless.
- **The relay stores nothing.** It holds no client secret (public PKCE client),
  keeps no state, sends `cache-control: no-store`, and only forwards the
  one-time `code` and `state` to your localhost callback.

## Why a relay exists

Slack requires an HTTPS redirect URI for a distributed app, and
`http://localhost` is not HTTPS. The relay is a minimal HTTPS-to-localhost bounce
so Slack has a valid redirect target. It carries no secrets and performs no token
exchange.

## Worst case if the relay is compromised

PKCE protects an honest login, but it does not protect a flow an attacker starts
themselves. A compromised relay controls a registered redirect, so the residual
risk is consent-phishing amplification:

1. An attacker crafts a Slack consent URL for the public client with their own
   PKCE challenge and the relay as the redirect.
2. They trick a user into approving the Slack consent screen.
3. Slack sends the code to the relay, which the attacker controls.
4. The attacker exchanges it with their own verifier and obtains that user's
   token.

Bounds on the damage:

- The scopes are read-only (no write, no admin).
- It requires the user to approve an OAuth consent they did not start.
- Tokens are revocable, and `auth logout` revokes server-side.
- Already-authenticated users are unaffected; their tokens live locally and
  refresh directly with Slack.

## Credential handling and prompt injection

Slack content is untrusted input. agent-slack treats it as data, never as
instructions, and its own commands never print the token (`auth status` and every
command emit scopes, ids, and booleans, not the secret).

The honest limit: once an operator authorizes the app, the token is usable by
whatever runs as that operator. Storing it in the Keychain encrypts it at rest
and protects it from other users, other apps, and offline disk theft, but it does
not sandbox the token from the authorized process itself. So an AI agent that
follows a prompt injection hidden in Slack content and has arbitrary shell plus
network egress could read the token and send it out. This is inherent to any
agentic tool that holds a usable credential, not specific to agent-slack.

What bounds and reduces that risk:

- **Read-only, least-privilege scopes**: a leaked token can only read what the
  operator can already read. No write, admin, or destructive actions.
- **The CLI never emits the token**, so an agent limited to agent-slack commands
  cannot extract it; exfiltration requires going around the CLI.
- **Token rotation** expires an access token in about 12 hours (revoke to also
  invalidate the refresh token).
- **Operator controls** that live outside this project: do not let an agent
  execute instructions found in Slack content, and run it without arbitrary shell
  or network egress when handling untrusted content.

## Hardening

- The relay pages send a strict `Content-Security-Policy` with a per-request
  nonce and `referrer-policy: no-referrer`. Injected scripts or styles without
  the nonce cannot run.
- The relay is stateless and secret-free by design, so a compromise leaks
  nothing at rest.

## Remove the relay from the path

To avoid the shared relay entirely, register a `http://localhost` redirect URI on
the Slack app and log in against it directly:

```bash
agent-slack auth login \
  --redirect-uri http://localhost:45454/oauth/slack/callback
```

Slack only redirects to registered URIs, so this requires the localhost URI to be
registered on the app you use (the bundled app, or your own app via
`--client-id`). With a direct localhost redirect, the code goes straight from
Slack to your machine and no hosted component is in the path.
